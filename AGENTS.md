## Scope

This subtree defines and eventually implements the adaptive experience compiler, content graph, context resolver, A/Z suite, Request Mirror Lab, and AMTECH landing-page generation workflow.

## Core rules

1. Canonical content is complete, accessible, indexable, and useful without adaptation.
2. Runtime output is selected from finite reviewed content/design grammar unless a separately validated generative path is explicitly enabled.
3. No raw IP persistence, browser fingerprinting, identity stitching, sensitive-trait inference, or hidden household/wealth/lifestyle prediction.
4. Synthetic persona dimensions are benchmark labels only unless directly and knowingly supplied by the tester.
5. Claims cannot exceed evidence level.
6. Search crawlers and people receive the same primary topic, facts, price, limitations, and proof level.
7. Unknown features are discarded, not dynamically minted.
8. Any resolver error, uncertainty, version mismatch, timeout, or policy violation returns the canonical baseline.
9. HRR, graph scoring, ANN, Zig/WASM, and generative UI must beat simpler measured baselines before adoption.
10. Classical HRR/VSA is not quantum computing.

## Compiler discipline

Treat the website as a typed program:

```text
approved knowledge + offer + evidence + design grammar + route policy
-> typed intermediate representation
-> canonical pages + content slices + graph + vectors + structured data
-> validation
-> deployable artifacts
```

Required properties:

- deterministic builds;
- schema validation;
- stable IDs;
- checksums and versions;
- explicit dependencies;
- generated-file headers;
- reproducible fixtures;
- no hand-editing generated artifacts;
- build failure on claim/evidence/design contradictions.

## Landing-page work

For each page or page family:

- declare audience, owner question, offer, evidence ceiling, canonical route, CTA, and experiment role;
- identify which modules are fixed and which may vary;
- use current AMTECH tokens and component grammar;
- preserve Start Free / Managed from $400 product truth where relevant;
- show concrete work, result, approval, proof, and next action;
- avoid generic AI language and ornamental dashboards;
- add validation fixtures before claiming the page family is supported.

## Design collaboration

The graphic designer and framework must share one contract:

- design tokens and semantic color roles;
- component states and responsive constraints;
- layout grammar and reserved dimensions;
- visual evidence/provenance treatment;
- motion and reduced-motion behavior;
- accessibility and contrast;
- image/art direction metadata;
- permitted variation axes;
- forbidden combinations.

The resolver may choose among approved visual treatments. It may not invent a new brand system per visitor.

## Agentic and generative UI

Prefer interfaces that expose:

- current goal;
- work state;
- tool/action state;
- required owner decision;
- result/proof;
- recovery or rollback.

Use generated or streamed components only when they improve task comprehension or action. Chat remains one input/output surface, not the whole product.

## Validation

Every substantive implementation must report:

- commands run;
- fixtures and versions;
- correctness/relevance/performance results;
- privacy, security, SEO, and accessibility findings;
- comparison with rules/facet/static baselines;
- failures and fallbacks;
- validation not run.

Phase status must never advance from documentation alone.
