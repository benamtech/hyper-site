## Scope

This subtree is an agent-first hyper-targeted publishing compiler and vector information-server research project. It is not production accepted.

`site-manifest.yaml` is the single declarative corpus authority. Request Mirror, UI fixtures, geometry fixtures, and future field pages are profiles inside that manifest and compile through the same path.

## Core invariants

1. Namespace, symbol version, dimensions, axes, feature atoms, explicit context cases, and page prototype geometry are compiled before content, links, UI, resolver artifacts, or HTML.
2. Changing namespace or symbol version must change actual role/filler symbols.
3. Every declared prototype must survive geometry, `SiteSource`, `PageIR`, packed IR, agent context, and any authoritative retrieval path.
4. HTML, Markdown, sitemaps, packed vectors, and UI plans are emissions, never canonical source truth.
5. Agents propose typed `AgentPageProposal` objects. They do not directly publish HTML, routes, claims, or indexable pages.
6. Every proposal enters `noindex` at `research` and requires sourced weighted coverage contexts.
7. Proposal admission requires computed positive marginal coverage against the current corpus; the prose hypothesis is rationale only.
8. CSI is a batch/split comparison arm. Do not use CSI singleton information as the sole proposal gate.
9. Claims cannot exceed evidence level.
10. Synthetic dimensions, expected IDs, scale fixtures, and forced zero-match cases are test plumbing, not SEO proof.
11. No raw IP persistence, fingerprinting, identity stitching, private-history ingestion, sensitive-trait inference, or named-person profiling.
12. Crawlers and people receive the same primary topic, facts, price, limitations, proof, and links.
13. Resolver error, low confidence, version mismatch, timeout, or policy failure returns the complete baseline.
14. HRR, graph scoring, learned embeddings, ANN, Zig/Wasm, generated UI, and agent harnesses must beat simpler baselines before adoption.
15. Classical HRR/VSA is not quantum computing.
16. Synthetic 2,000-page compilation must never be described as reviewed, deployed, indexed, or revenue-producing pages.

## Compiler path

```text
approved source + independently sourced context cases
-> unified manifest
-> namespace/version-bound symbols
-> all-prototype hyper-vector space
-> computed coverage + neighbor suggestions + UI requirements
-> evidence-bounded semantic modules
-> SiteSource with all prototypes
-> PageIR + packed prototype arrays + CSR page graph
-> deterministic HTML/schema/sitemap/instruction/UI/agent emissions
-> gated field publication
```

Required properties:

- deterministic builds and source ordering;
- declared axes and governed values;
- stable IDs, namespace, symbol version, compiler version, and hashes;
- all-prototype packed parity;
- explicit primary-prototype semantics before field publication;
- evidence/reference validation;
- profile/page consistency;
- dependency-bounded invalidation;
- no hand-editing generated artifacts;
- build failure on claim, evidence, geometry, profile, design, route, dependency, or coverage contradictions.

## Agent-generated page work

Use `buildHyperAwareAgentContext()` before proposing content. It must expose all existing page prototypes, vector identity/hash, nearby pages, information objects, design capabilities, generation rules, coverage policy, and publication boundary.

A proposal must include:

- independently sourced weighted context cases;
- target page prototype set;
- canonical question;
- distinct information/utility object;
- evidence and claims;
- semantic modules;
- graph/path rationale;
- design capabilities;
- generation provenance.

Insert proposals only through `applyAgentPageProposal()` and recompile the complete manifest. Passing computed coverage is necessary, not sufficient: external relevance, utility, evidence, graph, UI, crawl, and human-review gates remain.

## UI metaprogramming

The UI pass consumes compiled semantic modules, page matrix, prototypes, capabilities, browser policy, and publication state. The renderer may derive a clean professional design-system superset, but it cannot alter:

- prototypes or vector identity;
- canonical questions and semantic content;
- claims, evidence, and information objects;
- approved graph edges;
- required layout roles/capabilities;
- profile membership, indexability, or publication state;
- metadata, structured data, instruction projections, accessibility, or JS-disabled completeness.

## Public AI Employee interaction

Static pages and public employee execution are separate planes. Do not represent generic chat, R3F, or an instruction pointer as completed employee work.

A future public task surface must follow:

- `../../mvp-build/docs/public-interaction-standard.md`;
- `../../mvp-build/docs/ux/05-generative-ui-frontier.md`;
- secure deterministic controls for secrets/risky inputs;
- session streaming;
- typed artifact state;
- approval, proof, fallback, and recovery.

## Memory protocol

`memory/` is append-only narrative state.

- `memory/MEMORY.md` is the cumulative TOON-style index and current-state map.
- Every substantial session creates a new timestamped file: `YYYY-MM-DD-HHMM-<slug>.md`.
- Never overwrite, repurpose, compress away, or “refresh” an earlier timestamped handoff.
- Update `MEMORY.md` by adding the new handoff newest-first and reconciling current facts.
- Use file-reference-heavy, token-efficient entries; do not duplicate source or validation reports.
- TOON-style structure is a context encoding convention, not canonical application data.
- Record exact branch, commit, CI run, artifacts, status, unresolved risks, and next move.

## Field publication

A page may enter a field cohort only after relevance, information gain, evidence, distinctness, path, UI, accessibility, browser, crawl, canonical, lifecycle-cost, and publication gates pass.

Do not use a fixed 90-day folklore rule as acceptance. Use predeclared observation windows and record crawl, index, query, citation, lead, revenue, gross-profit, and lifecycle-return evidence.

## Validation reporting

Every substantive implementation records:

- exact commands, CI runs, commits, fixtures, versions, hashes, and profiles;
- correctness, relevance, performance, SEO, accessibility, privacy, and security findings;
- comparisons with lexical, learned semantic, facets, graph, HRR, hybrid, static, and neutral-renderer controls;
- facility-location and CSI assumptions when used;
- failures, fallbacks, and validation not run.

Phase status never advances from documentation or synthetic scale alone.
