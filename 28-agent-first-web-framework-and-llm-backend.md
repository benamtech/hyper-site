# 28 — Agent-First Web Framework and LLM Backend

Status: canonical architecture reframe; backend pass implemented as a tested near-alpha slice, full live orchestration and deployment pending  
Updated: 2026-07-18  
Controlling source: `reference/src/pcn-emitter.ts`, `reference/src/articleir-parser.ts`, `reference/src/unfolder.ts`, `reference/src/page-backend.ts`

## 1. Product identity

Hyper Site is one agent-first web framework with a mega-site generation mode. It is not a bulk-content product with a renderer attached.

The complete product stack is:

```text
agent workspace and project memory
-> business truth, evidence, brand, assets, approvals, outcomes
-> framework frontend
   ontology, typed graph, page coordinates, site program, design authority
-> generative backend ABI
   PCN -> provider -> ArticleIR
-> deterministic assembly
   Markdown, JSON-LD, HTML, sitemap state, asset references
-> web-framework delivery
   preview, dev server, publisher, CDN, incremental rebuilds
-> outcome loop
   browser, accessibility, indexing, ranking, conversion, maintenance evidence
-> next checkpointed agent action
```

The content writer, researcher, information architect, layout designer, renderer, publisher, and maintainer are top-level agent capabilities operating over one durable framework workspace. They do not become independent sources of truth or parallel publication pipelines.

Mega-site generation is one scale mode. The same framework must produce a coherent five-page site, maintain a 100-page site, and selectively rebuild a 10,000-page site.

## 2. Compiler architecture

Hyper Site already has substantial frontend and middle-end infrastructure:

```text
ProjectInput
-> ApprovedOntology
-> typed relations and constraints
-> selected opportunity/page regions
-> CandidatePageSeed / PageCoordinate
-> SiteGenerationPlan
-> approved DesignAuthority
```

The missing pass was the generative backend that lowers those semantic contracts into real prose and then into deterministic web artifacts.

The canonical lowering path is now:

```text
CandidatePageSeed
+ ApprovedOntology
+ EvidenceLedger
+ SourceLedger
+ DesignAuthority
+ InternalLinkPlan
+ section/schema/style policy
-> Page Contract Notation (PCN)
-> LLM backend call
-> raw ArticleIR
-> strict ArticleIR parser
-> accepted ArticleIR
-> deterministic unfolder
-> Markdown + JSON-LD + HTML + sitemap state
```

The LLM is a backend target. It may optimize prose inside the supplied contract. It does not choose the page's existence, intent, evidence universe, section set, word allocation, internal-link targets, schema type, design authority, canonical URL, or publication state.

## 3. Page Contract Notation

`reference/src/pcn-emitter.ts` serializes compiler state into a compact deterministic contract.

PCN contains:

- page identity, route, canonical URL, question, intent, schema, voice, and compiler hashes;
- typed entities using the current service/problem/tool/outcome/context vocabulary;
- ontology relations expressed as causal or dependency edges;
- mechanism steps and explicit constraints;
- evidence IDs, source IDs, statements, source URLs, and bounded excerpts;
- ordered sections, headings, target word counts, and required evidence per section;
- mandatory and forbidden prose patterns;
- exact internal links and anchor text;
- deterministic schema fields.

`emitPageContractFromCompilerState` connects PCN to the real `CandidatePageSeed`, `ApprovedOntology`, project ledgers, and approved design authority. PCN is not generated by another model and is not a prompt template reconstructed from conversational memory.

## 4. ArticleIR

`reference/src/articleir-parser.ts` defines the accepted output ABI for the generative backend.

The parser rejects output when:

- any required section is absent;
- an unknown section is added;
- the declared section weight differs from PCN;
- actual section length falls outside the configured tolerance;
- a required evidence ID is missing;
- an unknown evidence ID is cited;
- a forbidden pattern appears;
- a mandatory pattern is absent;
- a planned link is missing, altered, or invented;
- the schema type or fixed schema fields drift;
- the output contains duplicate blocks or uncontrolled text outside ArticleIR.

ArticleIR is accepted compiler input only after these deterministic checks pass. A model's claim that its output is valid has no authority.

## 5. Deterministic unfolder

`reference/src/unfolder.ts` owns final syntax.

The unfolder emits:

- human-readable Markdown;
- evidence footnotes resolved through the evidence ledger;
- internal links from the planned link graph;
- JSON-LD from the fixed schema contract;
- canonical static HTML with metadata and shared CSS;
- noindex or indexable robots state supplied by the caller;
- a sitemap entry only for explicitly indexable output;
- stable Markdown, JSON-LD, HTML, and artifact hashes.

The model does not emit HTML, CSS, JSON-LD, canonical tags, robots policy, or sitemap XML. Layout and visual behavior remain owned by the approved design authority and deterministic renderer.

## 6. Batch backend

`reference/src/page-backend.ts` provides the first real backend runner:

- a physical provider request contains no more than 25 PCN contracts;
- the provider returns one JSON object containing one ArticleIR string per requested page;
- the existing provider's bounded repair loop receives exact parser failures;
- accepted ArticleIR is immediately unfolded into real static artifacts;
- a bounded worker pool can dispatch multiple 25-page batches with a default concurrency of 10;
- output order and batch/result hashes remain deterministic.

This removes the architectural fiction that a successful provider call or generated draft object is already a page.

## 7. Agent-first operating model

The agent loop operates over versioned workspace state:

```text
observe approved workspace and requested outcome
-> select a bounded tool/action
-> produce or revise a typed artifact
-> validate the artifact
-> commit an immutable checkpoint
-> render or publish when authorized
-> observe measured outcomes
-> invalidate only affected dependents
-> repeat or stop under explicit budget and policy
```

Representative agent capabilities:

| Capability | Framework authority |
|---|---|
| business researcher | proposes source-bound facts and evidence records |
| ontology architect | proposes typed entities and relations |
| information architect | proposes pages, paths, and page-existence records |
| content writer | executes PCN and returns ArticleIR |
| layout designer | proposes design authority, templates, components, and asset briefs |
| renderer | deterministic assembly from accepted IR and design authority |
| publisher | deploys already accepted artifacts and records deployment identity |
| maintainer | changes sources, components, pages, and graph state through dependency invalidation |
| outcome analyst | imports measurements and proposes reweighting or retirement actions |

No capability gets a private shadow database or direct unvalidated publication path.

## 8. Web-framework surface

The backend pass is necessary but not sufficient. Near-alpha still requires normal framework ergonomics:

- one human-readable project configuration;
- local development server and preview;
- component and layout authoring;
- asset pipeline;
- route and internal-link inspection;
- cold and incremental builds;
- one-command deployment;
- static hosting and CDN compatibility;
- accessibility and browser checks;
- source maps or equivalent artifact provenance;
- page, family, component, token, source, and ontology invalidation;
- rollback and checkpoint recovery.

Hyper Site is not proven as a web framework until these are compared against at least one ordinary static or SSR framework on the same real fixture.

## 9. Scale modes

```text
starter mode: 5 reviewed core pages
site mode: 25–500 real pages
mega-site mode: 1,000–10,000+ pages
maintenance mode: selective changes over any accepted site
```

Each tier uses the same PCN, ArticleIR, unfolder, renderer, validation, checkpoint, and publication concepts. Scale may change batching, provider choice, cache policy, embedding/index implementation, and deployment target. It may not bypass semantic or artifact acceptance.

## 10. Critical path and temporary freeze

Until the first real five-page site is emitted and deployed, changes should concentrate on:

```text
pcn-emitter.ts
articleir-parser.ts
unfolder.ts
page-backend.ts
page-generation/provider dispatch
production or agent-site orchestration
publisher/deploy scripts
real case fixtures and browser tests
```

The following remain available as existing code or comparison arms but are frozen from speculative extension during the first live-site sprint:

```text
wasm.ts
zig/
ui-r3f/
context-corpus.ts
acceleration-decision.ts
opportunity-space-optimized.ts algorithm research
```

This is a prioritization rule, not a claim that Wasm, HRR, BM25, isotonic regression, R3F, or native kernels are inherently unsafe or permanently useless. They may be promoted when a measured critical-path workload and a simpler baseline justify them.

## 11. Immediate sprint

### Sprint 0 — first real site artifact

1. Connect the approved compiler plan to `emitPageContractFromCompilerState`.
2. Replace the current synthetic `GeneratedPageDraft` acceptance path with PCN/ArticleIR lowering for one real fixture.
3. Persist PCN, raw provider response, ArticleIR, and unfolded artifacts per batch.
4. Emit five coherent core/supporting pages under one approved design authority.
5. Run browser, accessibility, link, evidence, canonical, and noindex checks.
6. Deploy the noindex review site through one publisher adapter.

Definition of done:

```text
one real source-bound five-page site
+ accepted PCN and ArticleIR artifacts
+ deterministic Markdown/JSON-LD/HTML
+ browser-accessible review URL
+ exact build/deployment identity
```

### Sprint 1 — 100-page measured cohort

- bounded parallel dispatch with backpressure;
- durable per-batch checkpoint storage;
- real corpus novelty/information-gain experiment with a named baseline;
- noindex deployment and sitemap/URL submission where appropriate;
- cost, latency, rejection, retry, output, accessibility, and operator metrics.

### Sprint 2 — framework comparison and scale

- equivalent ordinary-framework implementation;
- cold and incremental build comparison;
- 1,000-page throughput test only after 5/25/100 real tiers pass;
- publication approval remains separate from generation success.

### Sprint 3 — measured feedback loop

- import indexing, ranking, UX, conversion, and maintenance observations;
- attach observations to page, evidence, template, and graph identities;
- compare reweighting/retirement strategies against simpler baselines;
- rebuild only justified dependents.

## 12. Explicit nonclaims

The current backend slice does not prove:

- a live provider has returned an accepted ArticleIR batch;
- the full production orchestrator uses PCN/ArticleIR for every page;
- a publisher has deployed the new backend output;
- a live review URL exists;
- the generated prose is attractive, accurate, useful, or rank-worthy on a real case;
- dense embeddings or information-gain scores improve outcomes;
- 10-way concurrency reaches any claimed pages-per-hour rate;
- Cloudflare, R2, S3, Search Console, or another deployment/measurement integration is complete;
- indexing, ranking, traffic, conversion, revenue, or framework superiority.

The next architectural gate is not another synthetic page-count benchmark. It is one real source-bound site lowered through PCN and ArticleIR into deployed static artifacts.
