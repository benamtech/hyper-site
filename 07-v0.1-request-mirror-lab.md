# 07 — v0.1 Request Mirror Lab

Status: immediate prototype specification
Purpose: validate request observability, transparent shape construction, HRR matching, edge latency, and rendering behavior before applying the framework to the AMTECH marketing site

## Product decision

v0.1 is a deliberately plain, Web-1-looking diagnostic website.

It is **not** the public AMTECH homepage, not a customer profile, and not an indexable hyper-personalized landing-page network.

It answers:

> What can this request and browser actually reveal, which facts are direct versus inferred, what context shape do those facts produce, and which finite page slice does the resolver select?

The entire point is visibility. The visitor sees every field used, its source, its retention, its weight, and the exact match result.

## Why this is the fastest valid experiment

A polished marketing page would hide whether the core mechanism works. The Request Mirror Lab exposes:

- server-visible request data;
- Cloudflare edge metadata;
- browser-visible data after an explicit action;
- explicit/simulated traits supplied through test controls;
- safe non-sensitive inferences;
- normalized feature vocabulary;
- weighted HRR context vector summary;
- candidate slice scores;
- graph and business prior contribution;
- fallback/confidence decision;
- edge and render timing.

It produces falsifiable artifacts instead of a persuasive demo.

## Indexing decision

v0.1 should be deployed publicly for testing but served with:

```text
X-Robots-Tag: noindex, nofollow
Cache-Control: private, no-store
```

Do not ask Google to index the personalized Request Mirror output.

Reasons:

- Googlebot sees its own request shape, not a human tester's shape;
- Cloudflare Pages preview deployments are `noindex` by default;
- Google recrawl/index processing is not an immediate test loop;
- dynamic testing is allowed only when it is not cloaking and the canonical content remains equivalent;
- request-specific diagnostic data should not become searchable.

SEO validation occurs later on a separate canonical generic page whose initial HTML explains the experiment. Use Search Console URL Inspection and rendered HTML comparison after the canonical page is production-ready.

## Deployment recommendation

### Fastest path

Deploy a standalone Cloudflare Worker on a public versioned or aliased preview URL, then attach a custom test subdomain when the first fixtures pass.

Suggested progression:

1. local Wrangler dev for unit/fixture work;
2. Worker preview URL for real `request.cf` and cross-device tests;
3. `holo-lab.<approved-domain>` custom subdomain for stable browser/load testing;
4. keep `noindex` throughout v0.1;
5. only the later canonical explanatory page is submitted to Search Console.

A Worker preview is superior to the Cloudflare dashboard playground because `request.cf` is not available in the dashboard/playground preview context.

### Next.js relationship

The fixed long-term stack remains Next.js 16+, but the first edge-observability proof should be a thin Cloudflare Worker because it gives immediate access to Cloudflare request metadata and isolates the resolver.

Two acceptable v0.1 modes:

- **Worker-native diagnostic HTML:** fastest, minimal dependencies, best for request/latency validation.
- **Worker resolver -> Next.js 16 page:** validates the final topology after the native lab passes.

Do not block the first experiment on the full Contentlayer/MDX/PPR application.

## Page appearance

Make it look intentionally early-web and technical:

- system serif or monospace font;
- white background;
- blue underlined links;
- plain `<h1>`, `<fieldset>`, `<table>`, `<dl>`, `<pre>`, `<details>`;
- no animation;
- no React client shell for the first server response;
- one small script loaded only after the visitor chooses to reveal browser signals;
- source and build version printed in the footer.

Suggested title:

```text
Holographic Request Mirror v0.1
```

Suggested intro:

```text
This page shows what this request reveals, what your browser can reveal with permission,
and how those facts are converted into a temporary matching shape. Nothing here is a
customer record. Uncheck any optional signal and recompute the match.
```

## Page sections

### 1. Request identity

Display non-secret request facts:

- request method;
- full test URL with sensitive query values redacted;
- host;
- pathname;
- normalized query keys;
- request timestamp;
- resolver POP/colo;
- generated correlation ID;
- experiment arm;
- content/vector/graph build version.

Do not display or store raw cookies, authorization headers, or security tokens.

### 2. Server-observed headers

Allowlisted display:

- `Accept`;
- `Accept-Encoding` / original client encoding when available;
- `Accept-Language`;
- `User-Agent`;
- low-entropy UA client hints;
- `Referer`, reduced to origin/category by default;
- `DNT` and `Sec-GPC` where present;
- broad fetch/navigation headers;
- protocol/TLS metadata supplied by Cloudflare.

Show absent/unsupported explicitly.

Never reflect arbitrary headers into HTML without escaping.

### 3. Cloudflare edge metadata

Cloudflare currently exposes request metadata that may include:

- country;
- region/region code;
- city;
- timezone;
- continent;
- postal/metro code;
- latitude/longitude;
- ASN and AS organization;
- colo;
- HTTP protocol;
- TLS version;
- approximate RTT/connection fields where available;
- bot-management fields only on plans where enabled.

v0.1 display policy:

- show country, region, city, timezone, continent, ASN/organization, colo, protocol, TLS;
- hide exact latitude, longitude, and postal code by default behind an explicit “show precision” action;
- never persist precise location;
- label all IP-derived fields as approximate and fallible;
- distinguish missing plan-dependent fields.

### 4. Browser signals — explicit reveal

Initial server render contains no active fingerprint collection.

Button:

```text
[Reveal browser-visible signals for this test]
```

After click, collect and display a bounded allowlist:

- viewport width/height bucket;
- screen width/height bucket;
- device-pixel-ratio bucket;
- browser timezone and offset;
- languages;
- color-scheme and reduced-motion preferences;
- pointer/hover/touch capability;
- online state;
- hardware-concurrency bucket;
- device-memory bucket where supported;
- connection effective type/downlink/save-data where supported;
- low-entropy UA data;
- cookie/storage availability;
- document visibility;
- page navigation timing summary.

Do not collect canvas, audio, WebGL renderer, font enumeration, installed plugins, battery, sensor, local-network, or other fingerprint surfaces.

High-entropy UA Client Hints require a second explicit control and remain disabled by default:

```text
[Request high-entropy browser details]
```

The UI must explain that these fields can increase identifiability and may not be available across browsers.

### 5. Explicit test traits

To test combinations such as “family + dog + homeowner” without pretending they were silently discovered, provide a form:

```text
I am testing this page as:
[ ] homeowner
[ ] renter
[ ] business owner
[ ] family household
[ ] dog owner
[ ] painting contractor
[ ] commercial fleet operator
Region: [select]
Current need: [select]
```

These values are labeled `explicit_test_input` and weighted more strongly than passive signals.

Support reproducible signed test URLs for QA, for example:

```text
/lab?fixture=homeowner-family-dog-residential
/lab?fixture=ny-commercial-fleet
```

Fixture URLs remain `noindex` and use a finite allowlist. Arbitrary query strings cannot mint traits.

### 6. Inference ledger

Every inference row shows:

- inferred label;
- source facts;
- deterministic rule/model version;
- confidence;
- whether the inference enters the HRR shape;
- how to disable it.

Allowed v0.1 inference examples:

- `device_class.mobile` from coarse viewport/UA signals;
- `local_daypart.evening` from coarse timezone;
- `network_class.business_or_cloud` from ASN category, with low confidence;
- `intent.residential_estimate` from explicit fixture/query/route;
- `source.organic_search` from referrer category.

Prohibited v0.1 inferences:

- family status from device/location;
- pet ownership from browsing/device data;
- home ownership from neighborhood/income proxies;
- race, religion, politics, health, sexuality, disability, union status, or other sensitive traits;
- creditworthiness, income, or vulnerability;
- named identity or employer/customer record resolution.

Family, dog, homeowner, and fleet-owner traits are valid only when explicitly supplied in the test fixture, consented first-party session, or a later lawful public-data experiment with separate review.

### 7. Normalized feature vector

Display a readable sparse representation, not 2,048 raw floats by default:

```text
route.lab                         weight 1.00
explicit.homeowner               weight 1.00
explicit.family_household        weight 1.00
explicit.dog_owner               weight 1.00
region.us_west                   weight 0.30
source.direct                    weight 0.20
device.mobile                    weight 0.20
```

Offer a downloadable/debug `<details>` section with:

- vector dimension;
- norm;
- artifact checksum;
- first/last small sample of float values;
- deterministic shape hash used only for this response, not tracking;
- JS versus WASM implementation;
- build and compute timings.

### 8. Candidate slices

Start with a small finite corpus:

1. `generic-baseline`;
2. `residential-family-dog`;
3. `commercial-fleet-ny`;
4. `contractor-estimate-evening`;
5. `technical-security-first`.

Each candidate is plain HTML/MDX content with typed facets and evidence level.

Display scoring:

```text
candidate                    HRR    graph   prior   penalty   final
residential-family-dog       .78    .62     .50     .00       .69
contractor-estimate-evening  .42    .55     .70     .00       .53
generic-baseline             .31    .80     .60     .00       .49
```

Show top-score threshold and top-two margin. If confidence fails, render generic baseline.

### 9. Materialized slice

Below the diagnostics, render the selected slice as a complete simple landing-page fragment.

The same page must offer:

- `View baseline`;
- `View selected slice`;
- `Compare source`;
- `Recompute without optional signals`;
- `Clear session data`.

No content is generated by an LLM at request time.

### 10. Timing and cache report

Display `Server-Timing` values:

```text
normalize
shape
cluster
score
policy
render
resolver-total
```

Also show:

- cold/warm indicator where measurable;
- cache status;
- artifact load status;
- JS/WASM fallback;
- remote calls count, expected `0` for matching.

## v0.1 shape pipeline

```text
request headers + request.cf
-> allowlist and redact
-> server feature vector
-> render diagnostic HTML immediately
-> optional browser reveal POST
-> explicit test traits POST
-> combine allowed features
-> weighted superposition of pre-bound vectors
-> normalize
-> flat exact search over tiny candidate corpus
-> graph/business prior lookup
-> hard eligibility + confidence gate
-> render selected finite slice
```

## Vector search decision

Do not use ruVector, Redis, pgvector, or HNSW in v0.1.

With five to perhaps hundreds of candidates, an in-memory flat dot-product scan is simpler, exactly measurable, easier to validate, and likely faster than network/database search. ruVector becomes a validation arm only when corpus size and measured latency justify approximate search.

## Zig/WASM scope

v0.1 includes:

- TypeScript reference core;
- Zig core with deterministic fixtures;
- pre-bound vectors generated at build time;
- hot-path weighted add, normalize, flat top-k dot product;
- request-time circular convolution disabled by default;
- runtime toggle between JS and WASM for benchmark comparison.

WASM is not required to serve the baseline. A WASM failure falls back to JS, then generic shape.

## Caching

Default diagnostic response:

```text
Cache-Control: private, no-store
```

Do not cache by shape hash.

Generated vector/content artifacts are immutable and cacheable. Later materialized public pages may cache by finite `variant_id`, but the request mirror itself should remain private/no-store.

Redis is not needed in v0.1. If session continuity is tested, use a short-lived opaque cookie and edge KV/Durable Object only after privacy/latency validation.

## Fast test plan

### Phase A — deterministic fixtures

- run server requests with signed fixtures;
- compare exact HTML, features, scores, and selected slice;
- run TypeScript/Zig golden tests;
- fuzz headers and query inputs;
- measure local Worker CPU time.

### Phase B — Cloudflare preview

- deploy to a Worker preview URL;
- verify `request.cf` fields from multiple devices/networks/VPN regions;
- use `curl -I` and `curl -s` with controlled headers;
- verify `X-Robots-Tag` and `Cache-Control`;
- collect `Server-Timing`;
- compare JS/WASM paths;
- verify no persistent identifier or raw IP appears in storage/log payloads.

Worker preview URLs currently do not provide normal Worker logs, so use bounded diagnostic output and/or deploy a stable test Worker/custom subdomain for observability.

### Phase C — browser matrix

Test:

- Chrome/Edge with low-entropy UA hints;
- Firefox/Safari where `NavigatorUAData` is absent;
- mobile/desktop;
- JS disabled;
- privacy/reduced-signal browser;
- consent declined/granted/withdrawn;
- high-entropy hints denied/unavailable.

### Phase D — stable custom test URL

Attach a custom test subdomain, keep `noindex`, and run:

- multi-region load tests;
- Cloudflare Worker analytics/logging;
- screenshot/HTML parity;
- external security/privacy review;
- human label test for match quality.

### Phase E — SEO shell experiment

Create a separate canonical generic explainer URL on the production domain:

```text
/holographic-hyper-targeting-lab
```

It contains a stable server-rendered explanation and embeds or links to the noindex mirror. Validate with:

- direct HTML/curl;
- Rich Results Test when structured data is present;
- Search Console live URL Inspection;
- request indexing only after parity and claim review.

Do not use a Google `site:` search as the primary verification; URL Inspection is more reliable and indexing is not immediate.

## v0.1 pass criteria

- every displayed fact has source/provenance and direct/inferred/explicit label;
- no prohibited fingerprint surface or sensitive inference;
- generic page complete with JS disabled;
- Cloudflare request metadata rendered accurately with missing-field handling;
- optional browser signals collected only after action and cleared on request;
- TypeScript and Zig scoring agree within the declared tolerance;
- selected slice matches all golden fixtures;
- malformed/unknown input falls back to generic;
- resolver compute p95 <= 10 ms warm and total overhead p95 <= 50 ms target on Cloudflare;
- no remote vector/database call;
- `noindex`, private/no-store, and no raw secret/cookie reflection;
- no visible flicker;
- all notes, anomalies, platform differences, and failed assumptions recorded in the research log.

## v0.1 non-goals

- conversion optimization;
- live AMTECH marketing personalization;
- indexing per-shape content;
- identifying a visitor;
- scraping public data about a named individual;
- customer/CRM enrichment;
- persistent behavioral profile;
- production recommendation infrastructure;
- arbitrary dynamic copy generation;
- declaring HRR superior before benchmark evidence.
