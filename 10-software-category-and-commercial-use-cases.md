# 10 — Software Category and Commercial Use Cases

Status: product-category definition
Updated: 2026-07-17

## Primary category

The software is an **adaptive experience compiler and edge decisioning runtime**.

A commercial shorthand is **Experience Materialization Platform**.

It ingests a company’s approved knowledge, offers, claims, proof, visual system, routes, audiences, channels, and experiment policy. It compiles them into a complete canonical website plus finite, explainable context-resolved landing experiences and campaign artifacts.

## What it combines

The product sits at the intersection of:

- headless content management;
- static-site and landing-page generation;
- adaptive hypermedia and personalized information retrieval;
- recommendation and decisioning engines;
- feature flags and experimentation;
- edge compute and request routing;
- knowledge graphs and programmatic SEO;
- agentic interaction surfaces;
- generative UI and component compilation;
- multi-channel content operations.

Existing tools usually solve one layer. This framework treats the whole online experience as one typed, testable program.

## What it is not

It is not primarily:

- a drag-and-drop page builder;
- an unconstrained prompt-to-website generator;
- a customer-data platform;
- an identity graph;
- a fingerprinting or surveillance product;
- a conventional A/B testing widget;
- an ad cloaking or doorway-page system;
- a chatbot embedded in a marketing site;
- a generic CMS theme.

## Core product modules

### 1. Company knowledge and evidence registry

Stores typed facts, offers, entities, claims, evidence levels, objections, tasks, proof objects, and approved CTAs.

### 2. Content and design compiler

Transforms the registry into canonical routes, landing-page families, component manifests, internal links, metadata, JSON-LD, channel outputs, and deterministic build artifacts.

### 3. Context resolver

Uses allowlisted request/session context to rank finite eligible experiences. It returns a bounded variant ID or the canonical baseline.

### 4. A/Z suite

Compares the complete baseline against continuous finite resolution. It measures relevance, user effort, conversion, trust, latency, accessibility, privacy, SEO parity, and truth preservation.

### 5. Distribution fabric

Publishes the same governed knowledge into websites, campaign pages, ad groups, email, social, sales enablement, AI-agent responses, MCP/ChatGPT apps, and structured APIs without independently rewriting every channel.

### 6. Agentic interaction runtime

Renders task-specific work states, approvals, forms, proofs, tools, and recovery paths when a conversational response is insufficient.

## What a normal company would buy it for

A company would use the platform to stop maintaining disconnected pages, campaigns, sales materials, and AI prompts by hand.

Typical workflow:

```text
company defines offer, proof, audience, objections, design system, and constraints once
-> compiler creates canonical site and approved page families
-> resolver presents the most relevant proof/order/interaction for a bounded context
-> distribution fabric emits campaign and agent-facing artifacts
-> A/Z suite measures which representations reduce effort and improve qualified outcomes
-> approved findings update the next versioned build
```

## Practical use cases

### SaaS company

- Compile role-specific pages for founder, engineer, operations, finance, and security evaluators.
- Vary proof depth and interaction mode without changing product facts.
- Generate campaign pages from one offer/evidence graph.
- Present architecture or integration modules only when relevant.

### Local or multi-location service business

- Produce canonical service/location pages only where distinct information exists.
- Resolve residential versus commercial examples, urgency, financing, or proof order from explicit route/campaign context.
- Keep prices, service areas, licenses, and testimonials synchronized.
- Generate ad landing pages without thin duplicate SEO pages.

### Agency

- Operate a governed multi-client landing-page factory.
- Encode each client’s brand, evidence, offers, and compliance rules.
- Generate page families and campaign assets with deterministic QA.
- Compare creative/offer structures across clients without sharing identities or data.

### Industrial or B2B company

- Adapt technical depth for engineers, procurement, executives, operators, and compliance teams.
- Compile product, application, industry, comparison, and proof pages from one entity graph.
- Present the correct specification, case study, approval flow, and contact action.

### Marketplace

- Compile category, seller, buyer, use-case, and trust pages.
- Resolve onboarding and proof modules by declared role and task.
- Keep policy, pricing, availability, and structured data consistent.

### Education or professional training

- Adapt explanation depth, examples, interaction, and navigation to explicit learner goals and demonstrated progress.
- Generate task-specific interfaces instead of forcing every question through chat.
- Preserve a complete accessible baseline.

### AI-native software company

- Generate product pages, documentation, onboarding, agent-tool interfaces, and MCP/ChatGPT app surfaces from shared schemas.
- Align web content with the actual tool and action graph.
- Expose work, approvals, telemetry, and proof as interface components.

## AMTECH-specific use

For AMTECH, the compiler must support at least:

- contractor and professional-office beachheads;
- Start Free, Managed from $400, and Workforce offers;
- estimate/follow-up, inbox, scheduling, bookkeeping, support, and software-operation examples;
- owner/operator, technical evaluator, security evaluator, and power-user proof depths;
- a zero-key, plain-text activation path;
- optional deep architecture and model/runtime information;
- canonical comparison against do-it-yourself agent stacks without turning the site into a feature marketplace.

## Value proposition

The platform reduces four forms of operational waste:

1. **Content duplication:** the same fact is rewritten and drifts across pages and channels.
2. **Design duplication:** every campaign recreates components and visual decisions.
3. **Experiment duplication:** each A/B test becomes a one-off implementation with weak governance.
4. **Interaction mismatch:** every user receives the same page or the same chat box regardless of task.

The product’s value is not infinite personalization. It is **governed variation at compiler scale**.

## Commercial packaging hypothesis

Possible product layers:

- **Compiler SDK:** schemas, content IR, page generation, structured data, validation.
- **Edge Resolver:** hosted or self-hosted decisioning runtime.
- **A/Z Suite:** experiments, diagnostics, guardrails, and offline optimization.
- **Design/Content Studio:** visual grammar, page-family authoring, previews, and approvals.
- **Distribution API:** channel-specific exports and agent-readable content.
- **Managed Experience Operations:** AMTECH or an agency maintains the knowledge, design, experiments, and releases.

## Category acceptance test

The category definition is useful only if an unrelated company can answer all of these clearly:

- What governed inputs do we provide?
- What deployable outputs do we receive?
- Which variations are allowed?
- Which facts and designs can never change automatically?
- How does the baseline work when resolution is off?
- How are outcomes measured?
- How can we reproduce or roll back any generated experience?

If those answers are unclear, the product is still an idea rather than usable software.
