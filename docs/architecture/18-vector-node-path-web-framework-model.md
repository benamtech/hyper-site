# 18 — Vector Node Paths and Public AI Employee Surfaces

Status: controlling conceptual and validation addendum; no new implementation claimed
Updated: 2026-07-17
Scope: `GTM-RESEARCH/website-framework/` and its relationship to `mvp-build/`

## Purpose

Define exactly how the high-dimensional compatibility research can become a practical web framework for:

- hundreds or thousands of narrowly useful canonical landing pages;
- stable internal paths between problem, workflow, proof, integration, and offer nodes;
- live public AI Employee task surfaces such as “create an estimate with AI” or “use AI with QuickBooks”;
- deterministic agent-assisted site construction without hidden visitor profiling, arbitrary runtime page generation, or thin Cartesian-product content.

This document extends the validation authority in files `04`, `05`, `12`, `13`, `15`, and `16`. It does not replace their mathematical, accessibility, privacy, search, or production gates.

## Reality first

The current repository implements a vector-annotated static publishing reference:

- one YAML manifest;
- typed feature atoms and one or more page prototypes;
- deterministic HRR/VSA vectors;
- simple vector-neighbor link suggestions;
- evidence-bounded semantic modules;
- `SiteSource -> PageIR` compilation;
- one packed vector per compiled page;
- CSR links and capability indexes;
- neutral and AMTECH static HTML/CSS emissions;
- an agent proposal schema with noindex research entry;
- a source-level UI renderer and optional noncanonical R3F projection.

It does not yet implement:

- a production corpus opportunity engine;
- measured marginal coverage during proposal acceptance;
- preservation of all page prototypes in packed `PageIR`;
- typed graph-edge semantics and path optimization;
- a first-class public AI Employee task-surface contract;
- LLM streaming, secure controls, typed work artifacts, approval, or proof inside this website-framework package;
- real search-distribution, conversion, or revenue validation.

The framework must be described according to this boundary.

# 1. Core vocabulary

## 1.1 Feature atom

A feature atom is one typed, provenance-bearing statement about a valuable context or page:

```text
industry: painting-contractor
problem: estimate-backlog
use_case: estimate-drafting
stage: implementation
proof: worked-estimate
constraint: owner-approval
surface: public-task-page
```

Atoms are not arbitrary keywords. Each atom needs:

- a declared axis;
- an approved or reviewed value;
- a source/provenance record;
- a role in page selection, content, pathing, or validation.

## 1.2 Vector chunk

A vector chunk is a coherent set of role-bound atoms representing one useful situation within the larger compatibility space.

```text
chunk = {
  target context atoms,
  demand/provenance,
  required answer,
  proof need,
  constraints,
  compatible node class
}
```

A vector chunk is not a hidden dossier about one person. It is a reusable model of a situation such as:

> A painting contractor owner is trying to turn field notes into an estimate, needs owner approval before sending, and wants a concrete workflow rather than general AI advice.

A page may support several chunks through multiple prototypes. Those prototypes represent different legitimate entry situations for the same canonical question.

## 1.3 Canonical node

A canonical node is a stable URL and semantic object with:

- one canonical question;
- one primary job or decision;
- one or more vector chunks/prototypes;
- evidence-bounded claims;
- at least one distinct information or utility object;
- typed graph relationships;
- a publication policy;
- a conversion or next-step role.

HTML is an emission of the node. The node is not the HTML file.

## 1.4 Node path

A node path is the ordered sequence of useful semantic transitions available to a prospect or agent.

```text
entry question
-> precise answer/workflow node
-> live utility or public employee task
-> proof/control node
-> Start Free employee creation
-> managed employee when operational value increases
```

A path is not a forced funnel. Every transition must be justified by the current question, completed work, or next decision.

## 1.5 Hyper-personalized

For this framework, hyper-personalized means:

1. the public corpus contains a page unusually compatible with a narrow but valuable situation; and
2. after explicit interaction, the page can adapt the task experience to facts the user deliberately supplies in the current session.

It does not mean:

- inferring protected or sensitive traits;
- fingerprinting;
- reconstructing Google’s private user profile;
- creating a unique indexable page per visitor;
- changing product truth, evidence, price, or approval boundaries.

# 2. The three-plane architecture

The framework must keep three planes separate.

## 2.1 Publication plane — offline corpus construction

```text
approved source + demand evidence
-> candidate vector chunks
-> candidate canonical nodes
-> compatibility and eligibility scoring
-> corpus coverage/distinctness optimization
-> reviewed node set
-> semantic modules and page plans
-> deterministic emissions
```

This plane decides which stable URLs deserve to exist.

It is where HRR/VSA, facets, lexical/semantic evidence, graph structure, facility location, duplicate analysis, lifecycle cost, and human review compete.

No visitor request creates a canonical node.

## 2.2 Navigation plane — stable node paths

```text
current canonical node
+ typed graph edges
+ current explicit task/result state
-> next useful canonical node or action
```

Required edge types should eventually distinguish relationships such as:

- broader question;
- narrower situation;
- prerequisite;
- next workflow step;
- integration dependency;
- proof/control explanation;
- comparison alternative;
- live task surface;
- Start Free transition;
- managed-service transition.

Current source stores untyped page IDs and simple vector-neighbor links. Therefore current internal links are research suggestions, not validated node paths.

## 2.3 Interaction plane — explicit-session AI Employee work

A public AI Employee surface is attached to a canonical node when doing the task is more useful than only explaining it.

```text
complete canonical HTML answer
-> explicit user task/input
-> deterministic secure controls where needed
-> session-level LLM/employee work
-> streamed owner-safe state
-> typed artifact or preview
-> approval when consequences require it
-> action/proof or safe handoff
```

The interaction plane follows the AMTECH Public Interaction Standard:

- conversation captures intent;
- secrets and risky data use deterministic secure controls;
- chat is not storage;
- meaningful work becomes a typed object;
- safety and approval grammar do not vary;
- the owner sees business movement, not internal model/runtime language.

The current website framework does not yet represent this contract. The product runtime contains related seams, but this compiler currently emits static content and optional visual interaction only.

# 3. Node classes for an AMTECH site

A production corpus should contain a bounded grammar of node classes.

## 3.1 Category and hub nodes

Examples:

- AI Employee for contractors;
- AI Employees for painting companies;
- AI office work for small business.

Purpose:

- define the category;
- aggregate graph authority;
- route to precise workflows;
- establish product truth and offer.

## 3.2 Problem and diagnostic nodes

Examples:

- why estimates are still being written after dinner;
- how missed calls become lost contractor leads;
- why invoice follow-up stalls.

Purpose:

- satisfy problem-aware queries;
- explain causes and operating consequences;
- route to concrete work nodes.

## 3.3 Workflow and solution nodes

Examples:

- how to create an estimate with AI;
- AI estimate follow-up for painting contractors;
- AI invoice reminder workflow with owner approval.

Purpose:

- explain the complete operating path;
- show inputs, states, artifacts, approval, recovery, and proof;
- optionally host a public task surface.

## 3.4 Integration nodes

Examples:

- how to use AI with QuickBooks;
- AI Employee for Gmail and customer follow-up;
- AI scheduling with Google Calendar.

Purpose:

- answer integration-specific intent;
- distinguish preparation, read-only work, approval, and consequential actions;
- expose secure-control and credential boundaries;
- route to supported employee capabilities without claiming unaccepted integrations.

## 3.5 Public utility and employee-task nodes

Examples:

- create an estimate draft from job notes;
- turn a customer email into a reviewed reply;
- organize job photos and assumptions into a proof card.

Purpose:

- let a prospect experience one bounded unit of useful employee work;
- produce a typed artifact rather than a generic chat answer;
- create a natural Start Free transition.

A public task node must still contain a complete static answer and explanation before JavaScript or interaction.

## 3.6 Proof, control, and trust nodes

Examples:

- how owner approval works;
- credential custody and scoped access;
- what proof an AI Employee leaves;
- recovery when an action fails.

Purpose:

- answer risk and technical evaluation intent;
- support every workflow and integration family;
- prevent vague autonomy claims.

## 3.7 Commercial transition nodes

Examples:

- Start Free;
- Managed AI Employee from $400;
- managed workforce for multiple roles or locations.

Purpose:

- connect demonstrated work to the correct commitment level;
- preserve one product category and one owner relationship;
- avoid low-cost chatbot-tier positioning.

# 4. How vector chunks create a page corpus

## 4.1 Candidate generation

Agents and researchers propose situation chunks from:

- real queries and query fan-out;
- contractor problems and workflow states;
- sales objections;
- onboarding questions;
- product capabilities and limitations;
- proof inventory;
- integration tasks;
- first-party work artifacts;
- competitor and public-discussion gaps.

Each chunk must state the concrete question or task it represents.

## 4.2 Node formation

Chunks are grouped into one canonical node only when they share:

- the same canonical question;
- compatible answer structure;
- the same product truth and evidence ceiling;
- a coherent information or utility object;
- a stable internal-link role.

Multiple prototypes may describe different entrances to that node. They must not be collapsed into one average vector if doing so hides distinct compatibility regions.

## 4.3 Corpus selection

A candidate node is selected only if it contributes positive marginal value after considering:

- demand weight;
- compatibility quality;
- existing-node coverage;
- distinct information gain;
- live utility value;
- graph/path contribution;
- evidence strength;
- conversion relevance;
- review, crawl, update, and operational cost.

“Zero search volume” is neither a pass nor a fail. A zero-volume candidate passes only when it is a credible valuable situation with distinct utility, graph value, or commercial research value.

## 4.4 Emission

Selected nodes emit:

- canonical HTML;
- metadata and schema;
- sitemap entries when approved;
- internal links;
- instruction projections;
- UI plans;
- optional public-task bootstrap and fallback contracts once implemented;
- analytics IDs that trace results back to the node and source chunk.

# 5. Two concrete AMTECH paths

## 5.1 “How to create an estimate with AI”

### Entry chunks

```text
industry: painting-contractor
problem: estimate-backlog
use_case: estimate-drafting
stage: implement
proof: worked-estimate
constraint: owner-approval
```

Related chunks may vary by field-note format, estimate complexity, role, or proof requirement while retaining the same canonical question.

### Canonical node

The page explains:

- what information the employee needs;
- how it converts notes/photos/rules into a draft;
- assumptions and missing information;
- owner review;
- customer-send boundary;
- follow-up and proof.

### Public employee surface

A limited public employee may:

- accept low-risk job notes and optional redacted examples;
- stream “reading -> preparing -> needs your say -> ready”;
- produce a typed estimate draft and assumptions list;
- avoid sending, payment, or customer actions;
- offer Start Free to preserve the work and connect the real business.

The static page remains complete when the employee surface is unavailable.

## 5.2 “How to use AI with QuickBooks”

### Entry chunks

```text
problem: bookkeeping-administration
use_case: quickbooks-assisted-work
stage: evaluate or implement
proof: workflow-and-control
constraint: credential-custody
surface: integration-page or public-task-page
```

### Canonical node

The page must distinguish:

- document preparation versus direct ledger mutation;
- read-only evidence versus write actions;
- secure credential collection;
- owner review and approval;
- supported versus unaccepted integration claims;
- proof and recovery.

### Public employee surface

A limited surface may work from deliberately uploaded/exported, nonsecret example data before any account connection. Authentication, provider credentials, and account IDs stay in deterministic secure controls. The LLM does not receive secrets.

# 6. Agentic developer operating procedure

An agent creating a site or page family should follow this order.

1. Read product truth, proof state, interaction standards, and browser/design authority.
2. Identify a real problem/task/query region.
3. Declare the target vector chunks and their source evidence.
4. Choose a node class and canonical question.
5. Compare against current nodes and calculate or explicitly mark uncomputed marginal coverage.
6. Define the distinct information object or public utility.
7. Define the graph role and typed next steps.
8. Decide whether a live employee task materially improves the page.
9. Bind every claim to evidence and every consequential action to approval policy.
10. Compose semantic modules and design capabilities.
11. Submit a noindex `AgentPageProposal`.
12. Compile the complete corpus and inspect neighbors, collisions, links, UI, and emissions.
13. Run node, path, interaction, search, accessibility, and commercial validation.
14. Publish only through a separate approval gate.

Until measured marginal coverage and public-task contracts exist in code, steps 5 and 8 require human review and must be labeled uncomputed or not implemented.

# 8. Pass vector

The framework passes this conceptual gate when:

- vector chunks represent reviewed situations, not inferred identities;
- symbol generation is namespaced and versioned;
- every page prototype survives through all compiler stages that claim to use it;
- a node exists because it contributes measured positive marginal value;
- internal links express typed useful transitions, not raw similarity alone;
- each canonical page has one stable question and distinct information or utility value;
- a public employee surface is attached only when it improves task completion;
- static HTML remains complete and search-safe without interaction;
- streamed work produces typed artifacts and follows secure-control and approval standards;
- public work naturally transitions to Start Free without a fake demo boundary;
- managed conversion occurs when capacity, connection, maintenance, or operational importance increases;
- real search and commercial evidence justify each scale step.

# 9. Fail vector

The framework fails when:

- a “vector chunk” is only a keyword bundle;
- vectors are computed after page selection and do not affect corpus decisions;
- secondary prototypes are dropped while documentation claims multi-region support;
- a page is selected because an agent wrote a marginal-coverage sentence rather than because coverage was measured;
- untyped cosine neighbors become production internal links without semantic review;
- zero-volume is treated as proof of opportunity;
- synthetic benchmark labels are reported as real relevance evidence;
- “semantic” vectors are token hashes but described as learned semantic understanding;
- runtime creates new canonical pages or identity-shaped cache keys;
- a public employee is generic chat without a typed task, artifact, approval, or proof;
- secrets enter the LLM stream;
- the canvas or interactive bundle contains essential canonical content;
- pages are numerous, fast, and deterministic but do not populate useful search results or create qualified business value.

# 10. Current code-review blockers before agent proposals

The deep review recorded in `validation/reports/2026-07-17-vector-node-path-deep-code-review.md` blocks the first real proposal cohort until disposition is explicit.

Highest-impact blockers:

1. HRR symbols currently ignore manifest namespace/version even though hashes include namespace.
2. Broad vector-space hashing remains source-order-sensitive outside the UI-specific canonical hash.
3. Multi-prototype geometry is retained in `CompiledHyperVectorSpace` but reduced to the primary prototype in `SiteSource`, `PageIR`, and packed vectors.
4. `marginal_coverage_hypothesis` is validated as nonempty text; no optimizer result gates proposal insertion.
5. Auto-links are untyped max-cosine neighbors with profile overlap, not validated node paths.
6. The current synthetic relevance benchmark makes expected pages and zero-match behavior largely true by construction.
7. UI archetype/component selection is driven primarily by semantic module kinds; vector geometry is retained as metadata but does not yet generally choose layout.
8. No first-class contract connects website nodes to public streaming employee work, secure controls, artifacts, approvals, or proof.

The next implementation pass should begin only after these are accepted as deliberate limitations or converted into bounded corrective work.
